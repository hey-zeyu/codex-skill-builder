# Codex 交互指南

本文件用于把按钮式、多选式的 skill 创建流程改成 Codex 适合的自然语言协作流程。

## 核心原则

Codex 不需要每一步都向用户弹选择题。更好的节奏是：

1. 先理解用户的目标。
2. 对新 skill 想法，先一次性列出用户需求画像问题，然后停下等用户回答。
3. 用户回答后，先整理还缺什么。
4. 给出两个简单技术方案和推荐。
5. 列出新 skill 的工作流程。
6. 停下来等用户说“生成”，再创建文件。
7. 用户不同意时，允许直接纠偏。

默认话术：

```markdown
我先按这个判断继续；如果你不同意，直接说要改哪里。
```

## 用户需求画像

在开始设计 skill 前，先列出用户当下可能的关键情况。它的作用类似 `skill-state.json` 的 onboarding：不是为了闲聊，而是为了知道这个 skill 应该服务什么人、什么输入、什么输出、什么约束。

重要节奏：输出画像问题后必须停下，等待用户回答。不要在同一条回复里继续输出洞察、技术方案或蓝图。

输出模板：

```markdown
为了把这个 skill 做准，我需要先了解你当下这几类情况：

1. <问题>？
   比如：<示例答案/选项>

2. <问题>？
   比如：<示例答案/选项>

...

如果你有初版方案或者其他想法都可上传和填写。
你可以一次性回答；不确定的地方写“不确定”。
```

用户回答后，再继续：

```markdown
我先整理出你的需求画像：
- 已知：
  - ...
- 还缺：
  - ...
- 默认假设：
  - ...

我理解你的核心需求和工作流是：
- 输入：
- 工作流：
  1. ...
  2. ...
- 输出：
- 使用场景：

实现这个 skill，我看到两个方案：

方案 A：<简单方案>
- 优点：
- 缺点：
- 适合：

方案 B：<增强方案>
- 优点：
- 缺点：
- 适合：

我的建议：先选方案 <A/B>，因为 <原因>。

新 skill 的工作流程：
1. ...
2. ...
3. ...

下一步如果你说“生成”，我就创建工作区预览版 skill，不安装。
```

如果用户已经提供了部分信息，改成：

```markdown
我先整理出你的需求画像：
- 已知：
  - ...
- 还缺：
  - ...

缺的信息不影响第一版时，我会先按默认假设继续。
```

按 skill 类型生成问题，不要所有 skill 都问同一套。

### 求职/职业类

```markdown
1. 你现在的职业/技能是什么？
   比如：运营、销售、设计、编程、数据、财会、外语、某行业经验等。

2. 你想转型的幅度？
   比如：完全跨行 / 用旧技能升级 / 不确定。

3. 学历 + 工作年限？

4. 每周能稳定投入多少小时学习？能坚持几个月？

5. 学习能力/自驱力自评？
   比如：容易坚持、需要监督、以前自学成功过什么。

6. 所在地区 + 目标薪资？
   比如：远程、想去的城市、薪资范围。

7. 转型紧急度？
   比如：在职慢慢转 / 急需尽快上岸。
```

### 内容/视频类

```markdown
1. 你主要做哪个平台？
   比如：抖音、小红书、B站、视频号、快手。

2. 用户会输入什么？
   比如：视频链接、文案、截图、本地视频、行业关键词。

3. 希望输出什么？
   比如：口播稿、分镜表、标题、拍摄清单、改编策略。

4. 目标受众是谁？
   比如：本地生活商家、求职人群、宝妈、企业老板。

5. 原创边界是什么？
   比如：只拆结构不照抄 / 可参考表达 / 必须完全原创。

6. 成功标准是什么？
   比如：能直接拍、提升完播、提高咨询、适合批量生产。
```

### 开发/工具类

```markdown
1. 用户会提供什么输入？
   比如：README、GitHub repo、本地代码、API 文档、命令行说明。

2. 期望生成什么输出？
   比如：SKILL.md、脚本、配置文件、测试用例、报告。

3. 运行环境是什么？
   比如：Windows、macOS、Linux、Node、Python、Docker。

4. 是否需要联网或密钥？
   比如：GitHub token、OpenAI key、数据库连接。

5. 谁会使用这个 skill？
   比如：只有自己、团队、开源用户。

6. 怎么判断它做得好？
   比如：生成文件可运行、测试通过、少问问题、输出稳定。
```

### API/集成类

```markdown
1. 要连接哪个服务？
   比如：飞书、Notion、GitHub、企业微信、某个内部系统。

2. 允许读还是允许写？
   比如：只查询 / 可创建记录 / 可删除或覆盖。

3. 是否有凭证和权限？
   比如：API key、OAuth、cookie、企业账号。

4. 调用频率和成本限制？
   比如：每天几次、批量多少、是否付费接口。

5. 数据隐私边界？
   比如：不能保存用户信息、不能上传原始文件。
```

## 主动洞察模板

当用户只说了模糊想法时，先输出：

```markdown
我理解你的核心需求和工作流是：
- 输入：
- 工作流：
  1. ...
  2. ...
- 输出：
- 使用场景：

我还想到你可能需要：
1. <潜在需求>：<原因>
2. <边缘情况>：<原因>
3. <暂缓项>：<为什么第一版先不做>

我的默认建议：
- 第一版保留：
- 第一版暂缓：
- 成功标准：
```

注意：整理完用户需求画像后，不要马上生成文件。也不要把“架构判断”“Skill 蓝图”“I/O 契约”作为独立板块展示给用户。Codex 可以在内部判断架构和输入输出边界，但对用户只展示“还缺什么”“两个技术方案”“新 skill 的工作流程”，最后停在“下一步如果你说‘生成’”。

示例：

```markdown
我先整理出你的需求画像：
- 已知：
  - 输入：多个 Excel 文件
  - 输出：清洗后的 Excel
  - 规则：空值固定规则模板、字段统一
- 还缺：
  - 字段映射模板的具体字段
  - 空值规则细节
- 默认假设：
  - 缺的信息不影响第一版，我会先用可改的默认模板。

实现这个 skill，我看到两个方案：

方案 A：只写操作说明
- 优点：简单，生成快。
- 缺点：每次清洗还要 Codex 临时写代码，稳定性一般。
- 适合：偶尔手工处理。

方案 B：说明 + 固定 Python 脚本
- 优点：流程稳定，可批量处理多个 Excel。
- 缺点：第一版文件会稍多一点。
- 适合：你这种固定规则模板。

我的建议：先选方案 B，因为你要多个文件、固定规则、输出新 Excel。

新 skill 的工作流程：
1. 读取多个 Excel 文件。
2. 检查字段名。
3. 按字段映射规则统一字段。
4. 按空值规则处理缺失值。
5. 输出到新目录。
6. 生成清洗摘要。

下一步如果你说“生成”，我就创建工作区预览版 skill，不安装。
```

## 技术方案模板

当任务涉及外部工具、API、文件处理、网络请求或脚本时，用简单语言解释方案。

```markdown
实现这个 skill，我看到两个方案：

方案 A：<简单方案>
- 优点：
- 缺点：
- 适合：

方案 B：<增强方案>
- 优点：
- 缺点：
- 适合：

我的建议：先选方案 <A/B>，因为 <原因>。
如果你没有特别偏好，我按推荐方案继续。
```

避免：

```markdown
使用 REST API 还是 GraphQL？
```

改成：

```markdown
方案 A：直接读取文件，简单但功能有限。
方案 B：连接在线服务，功能更强但需要密钥和网络。
```

## 作用域模板

除非用户明确要求安装，否则先生成预览副本：

```markdown
我先按这个作用域处理：
- 使用工具：Codex
- 安装方式：先生成 skill 文件夹，不自动安装
- 作用域：当前工作区预览版
- 后续如果确认可用，再移动到全局 skills 目录
```

如果用户要求安装到全局位置，先确认目标路径和是否覆盖已有 skill。

## 确认与纠偏

只在这些情况停下来等用户：

- 要安装到全局 skills 目录。
- 要覆盖或删除用户原文件。
- 要执行网络请求、登录、发布、上传、付费 API 调用。
- 用户需求互相冲突。
- 缺少不可推断的信息，例如 API key、目标平台、输入文件。

普通设计细节不要反复确认。先给默认方案，继续推进。

## 避免的反模式

| 反模式 | 问题 | 改法 |
|---|---|---|
| 每一步都问 | 打断执行节奏 | 默认推荐 + 允许纠偏 |
| 问技术术语 | 用户可能不懂 | 用简单语言解释优缺点 |
| 让用户判断复杂度 | 用户很难判断 | Codex 自己评估架构 |
| 缺少默认选择 | 用户负担大 | 明确推荐第一版做什么 |
| 未确认就安装/覆盖 | 风险高 | 风险操作前确认 |
